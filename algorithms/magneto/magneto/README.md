
## Подход 1: Linearized Table + Column Markers

Не кодируем каждый столбец отдельно, а **линеаризуем таблицу** в один текстовый “документ” с **маркерами границ колонок**, прогоняем его через энкодер и **извлекаем эмбеддинг каждой колонки**.

---

### 1.1. Линеаризация таблицы
Мы строим строку вида:
```
[TABLE] <table_name>
[COL_START] <col_name_1> [TYPE] <type_1> [VALUES] v11 | v12 | ... | v1N [COL_END]
[COL_START] <col_name_2> [TYPE] <type_2> [VALUES] v21 | v22 | ... | v2N [COL_END]
...
```
и тут
- `[COL_START]` и `[COL_END]` — **маркеры начала/конца колонки**.
- `[TYPE]` — вставка типа (date/numeric/text/…).
- `[VALUES]` — вставка сэмплов значений из столбца.
- `v_ij` — значения

---

### 1.2. Как разделить колонки
c Hugging Face
#### специальные маркеры
1) Выбираем строковые маркеры, которые точно выделяются:
   - `[COL_START]`, `[COL_END]`, `[TYPE]`, `[VALUES]`
2) Добавляем их в токенизатор как **special tokens**, чтобы они не разрезались на кусочки:
   - тогда токенизатор гарантирует, что `[COL_START]` будет одним токеном, а не набором `[`, `COL`, `_`, `START`, `]`.
3) При токенизации получаем `input_ids`.
4) Находим позиции этих токенов:
   - индекс токена `[COL_START]` = начало блока
   - индекс токена `[COL_END]` = конец блока

**Что именно извлекать из выхода модели:**
- берём выходные hidden states для токенов между `[COL_START]` и `[COL_END]` и делаем mean/max pooling.
- берём hidden state именно токена `[COL_START]` как “представитель” колонки (аналог [CLS] для блока).

---

#### маркеры как текстовые паттерны + offset mapping
1) оставить маркеры как текст в строке
2) токенизировать с `return_offsets_mapping=True`
3) offsets дают соответствие между токенами и символами исходной строки
4) мы заранее знаем **символьные позиции** `[COL_START]... [COL_END]` в строке и по offsets определяем, какие токены попали в диапазон

---

### 1.3. Как получить эмбеддинг каждой колонки
После того как мы прогнали линерализованный текст через энкодер, у нас есть выход:
- `last_hidden_state` размера `[seq_len, hidden_dim]` (условно, для одной таблицы)

Далее для каждой колонки получаем embedding:

**Marker pooling:**
- `E_col = hidden_state[pos_COL_START]`

**Span pooling:**
- `E_col = mean(hidden_state[pos_COL_START : pos_COL_END])`

`pos_COL_START/pos_COL_END` — индексы токенов маркеров.

---

### !Ограничение по длине
Трансформеры имеют лимит длины контекста. Поэтому нужно :
- ограничивать `n_samples` для каждого столбца
- ограничивать количество столбцов
- убирать длинные значения (truncation)

---

### 1.5. Почему это должно дать улучшение
В текущей реализации столбец кодируется изолированно: он “не знает”, какие ещё поля рядом.  
В table-aware режиме эмбеддинг колонки будет зависеть от соседних колонок:
- `date` рядом с `admission` vs `date` рядом с `birth` — это разные смыслы
- “контекст таблицы” помогает разрулить неоднозначные колонки

## Подход 2: Local Window Context (колонка + k соседей)
### Идея
Если вся таблица не помещается, кодируем не всю таблицу, а “окно” вокруг текущей колонки:
- центральная колонка: header + type + values
- `k` соседей слева и справа: header (или header + маленький сэмпл)


    реализовать выбор разных колонок, из рахзных частей
---

## Подход 3
Сохраняем текущие эмбеддинги колонок, но добавляем один общий вектор таблицы:
- `E_col` — как сейчас
- `E_table` — эмбеддинг таблицы по заголовкам/короткому описанию
- итог: `E'_col = concat(E_col, E_table)` или `E'_col = E_col + W * E_table`

---

## Подход 4: Разделение таблицы + агрегирование (если таблица большая)
### Идея
Линеаризация таблицы как в подходе 1, но разбиваем таблицу на чанки по колонкам:
- chunk_1: колонки 1..M
- chunk_2: колонки M+1..2M
- можно overlap между чанками, с наложением

Для каждой колонки получаем embedding из чанка, где она присутствует.  
Если колонка присутствует в нескольких чанках (overlap), агрегируем:
- mean / max по векторам
- или weighted (больше вес “центральному” чанку)
